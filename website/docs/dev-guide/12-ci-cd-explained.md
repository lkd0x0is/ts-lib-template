This project utilizes GitHub Actions for its Continuous Integration (CI) and Continuous Deployment/Delivery (CD) pipelines. These automated workflows help ensure code quality, streamline the release process, and keep the documentation website up-to-date.

All workflow definitions are located in the `.github/workflows/` directory.

## Overview of Workflows

There are three primary workflows configured in this template:

1. **`ci-checks.yml` (CI Checks):** Focuses on code quality and correctness.
2. **`release.yml` (Release):** Manages automated versioning and publishing of packages.
3. **`deploy-docs.yml` (Deploy Docs):** Builds and deploys the Docusaurus documentation website.

Let's break down each workflow.

### 1. `ci-checks.yml` - Continuous Integration Checks

* **Purpose:** To automatically verify the integrity and quality of the codebase on every push to `main` and every pull request targeting `main`.
* **Triggers:**
  * `on: push` (for branches: `main`)
  * `on: pull_request` (for branches: `main`)
* **Key Steps:**
    1. **Checkout Code:** The workflow checks out the latest commit from the branch or PR.
    2. **Set up Environment:**
        * Sets up Node.js (using the version specified in the workflow, ideally aligned with `.nvmrc`).
        * Sets up pnpm and enables caching for pnpm dependencies to speed up subsequent runs.
    3. **Install Dependencies:** Runs `pnpm install` to install all necessary dependencies for the monorepo.
    4. **Lint & Format Check:** Runs `pnpm lint` (e.g., using BiomeJS) to ensure code style and formatting consistency, and to catch linting errors.
    5. **Type Check:** Runs `pnpm typecheck` (e.g., `tsc --noEmit` for each package) to verify TypeScript type correctness across the project.
    6. **Run Tests:** Executes `pnpm test` (e.g., using Vitest) to run all unit and integration tests. Coverage reports might also be generated and uploaded as artifacts.
    7. **Build Packages:** Runs `pnpm build` to compile all library packages, ensuring they can be successfully built (e.g., into ESM/CJS formats with Rollup).
* **Outcome:** If any of these steps fail, the workflow run is marked as failed. For Pull Requests, this typically blocks merging until the issues are resolved.

### 2. `release.yml` - Automated Package Release

* **Purpose:** To automate the versioning of packages, generation of changelogs, and publishing of updated packages to the npm registry using Changesets.
* **Triggers:**
  * `on: push` (for branches: `main`)
    * Typically configured with a condition to only run if changeset files (`.changeset/*.md`) exist or were modified in the push to `main`.
* **Key Steps:**
    1. **Checkout Code & Setup:** Similar to `ci-checks.yml`, sets up the environment.
    2. **Install Dependencies:** `pnpm install`.
    3. **Create Release PR or Version & Publish (managed by `changeset-action` or similar):**
        * This step is often handled by a dedicated GitHub Action like `changeset-actions/release` or by manually scripting `pnpm changeset version` and `pnpm changeset publish`.
        * **`changeset version`**:
            * Reads all markdown files in the `.changeset/` directory.
            * Bumps the versions in the `package.json` files of the specified packages.
            * Updates `CHANGELOG.md` for these packages.
            * Commits these changes (version bumps, updated changelogs, deleted changeset files) back to the `main` branch.
            * Pushes the commit to `main`.
        * **`changeset publish`**:
            * Publishes the packages that have new versions to npm.
            * This step requires the `NPM_TOKEN` secret.
    4. **Create GitHub Release (Optional but common):**
        * After successful publishing, the workflow might create a GitHub Release for the tag generated by Changesets, often using the changelog content for the release notes.
* **Outcome:** Successfully updated packages are published to npm, `CHANGELOG.md` files are updated, and GitHub Releases are created.

### 3. `deploy-docs.yml` - Documentation Website Deployment

* **Purpose:** To automatically build and deploy the Docusaurus documentation website (typically to GitHub Pages).
* **Triggers:**
  * `on: push` (for branches: `main`)
* **Key Steps:**
    1. **Checkout Code & Setup:** Sets up the environment.
    2. **Install Dependencies:** `pnpm install`.
    3. **Generate API Docs (If not pre-committed):**
        * This step might be necessary if your API documentation markdown (`packages/*/docs-markdown/`) is not committed to the repository and needs to be generated on-the-fly before Docusaurus builds.
        * Example: `pnpm docs:generate:api` (this might iterate through packages or be a more sophisticated script).
    4. **Build Docusaurus Site:** Runs `pnpm docs:build` (which usually executes `pnpm --filter website build`). This generates the static site in `website/build/`.
    5. **Deploy to GitHub Pages:** Uses an action like `actions/deploy-pages` or a custom script to deploy the contents of `website/build/` to the `gh-pages` branch or directly to GitHub Pages if configured for GitHub Actions.
* **Outcome:** The documentation website is updated with the latest content from the `main` branch.

## Required Secrets

For the workflows to operate correctly, especially the `release.yml` workflow, you need to configure secrets in your GitHub repository settings (`Settings > Secrets and variables > Actions > Repository secrets`):

* **`NPM_TOKEN` (Required for publishing):**
  * An npm [access token](https://docs.npmjs.com/creating-and-viewing-access-tokens) with "Automation" or "Publish" permissions for your packages/scope.
  * This token is used by `changeset publish` (or `pnpm publish`) to authenticate with the npm registry.

* **`GH_TOKEN` or `GITHUB_TOKEN` (Usually provided by GitHub):**
  * The `GITHUB_TOKEN` is automatically available to workflows and has permissions scoped to the repository. It's often used for actions like committing version bumps back to the repo or creating GitHub Releases.
  * For cross-repository actions or if default permissions are insufficient (e.g., for `gh-pages` deployment if specific permissions are needed), you might use a Personal Access Token (PAT) stored as a secret, but this is less common for standard setups.

## Customizing Workflows

You can customize these workflows by editing the YAML files in `.github/workflows/`. Common customizations include:

* **Node.js Versions:** Modify the `node-version` matrix in `ci-checks.yml` to test against multiple Node.js versions.

    ```yaml
    strategy:
      matrix:
        node-version: [18.x, 20.x, 22.x] # Example
    steps:
    - uses: actions/setup-node@v4
      with:
        node-version: ${{ matrix.node-version }}
    ```

* **Test Scripts:** Change the command used to run tests if you have different test suites (e.g., unit vs. E2E with different commands).
* **Build Steps:** Add or modify steps related to building your packages or website.
* **Deployment Targets:** Change the deployment target for `deploy-docs.yml` if you're not using GitHub Pages (e.g., deploying to Netlify, Vercel, AWS S3).
* **Conditional Execution:** Add `if:` conditions to steps or jobs to control when they run (e.g., skip publishing for certain branches or tags).
* **Caching:** Fine-tune pnpm caching strategies to optimize workflow run times. The template usually includes basic pnpm caching.
* **Notifications:** Add steps to send notifications (e.g., to Slack or Discord) on workflow success or failure.

### Workflow Permissions

Ensure your workflows have the necessary [permissions](https://docs.github.com/en/actions/security-guides/automatic-token-authentication#permissions-for-the-github_token). For example, the `release.yml` workflow needs `contents: write` to push version bumps and `packages: write` if publishing to GitHub Packages Registry (though this template defaults to npm). The `deploy-docs.yml` needs `pages: write` and `id-token: write` if using GitHub Pages with OpenID Connect.

These are usually set at the workflow level or job level:

```yaml
permissions:
  contents: write # Allows committing back to the repo
  pull-requests: read
  # For GitHub Pages deployment:
  pages: write
  id-token: write
```

By understanding and customizing these CI/CD workflows, you can create a robust automated pipeline tailored to your project's specific needs.
